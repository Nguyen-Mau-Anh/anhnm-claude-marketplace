# BMAD Auto-Story Pipeline (Dashboard Edition)

You are the **team lead** orchestrating the BMAD implementation pipeline for story "$ARGUMENTS". Your job: create an agent team, spawn fresh teammates for each phase, and coordinate the full story lifecycle autonomously.

Each teammate is a fresh Claude Code session that invokes BMAD skills directly. You coordinate by reading artifact files between phases and applying convergence detection to prevent infinite fix loops.

**This is the dashboard edition** — you maintain a persistent state file and render an ASCII dashboard at every checkpoint so the user can see real-time pipeline progress.

---

## SETUP

1. **Parse story key** from "$ARGUMENTS"
   - If empty: auto-discover by reading `_bmad-output/implementation-artifacts/sprint-status.yaml` and finding the FIRST story with status "backlog" (top to bottom)
   - Store the resolved `{story-key}` for all phases

2. **Read sprint-status.yaml** at `_bmad-output/implementation-artifacts/sprint-status.yaml`
   - Find the story's current status
   - Determine the **starting phase**:

   | Current Status  | Start From                    |
   |-----------------|-------------------------------|
   | `backlog`       | Phase 1 (CREATE-STORY)        |
   | `ready-for-dev` | Phase 3 (DEV-STORY)           |
   | `in-progress`   | Phase 3 (DEV-STORY resume)    |
   | `review`        | Phase 4 (CODE-REVIEW)         |
   | `done`          | SKIP — story already complete |

3. **Locate story file** (if starting from Phase 3+): search `_bmad-output/implementation-artifacts/` for `{story-key}.md`

4. **Create team** called `auto-{story-key}`

5. **Initialize pipeline state file** at `_bmad-output/implementation-artifacts/pipeline-dashboard.yaml`:
   - Write the initial state structure (see STATE TRACKING section)
   - Set all phases to `pending`
   - Mark skipped phases: if starting from Phase 3, set `create` and `validate` to `skipped`. If starting from Phase 4, also set `dev`, `qa_prepare` to `skipped`.
   - Record `start_time` by running: `date -u +%Y-%m-%dT%H:%M:%SZ`
   - **Render the initial dashboard** (shows entry point and all phases)

## GLOBAL RULES

1. **Fresh teammate per phase** — Spawn a NEW teammate for each phase. Never reuse. This matches BMAD's "fresh context per workflow" design.
2. **YOLO mode** — Tell every teammate: "Choose YOLO mode (y) when the workflow asks for input to auto-complete all sections."
3. **Evaluate by reading files** — Between phases, YOU (the lead) read sprint-status.yaml and story files directly. Don't rely solely on teammate messages.
4. **Sequential by default, parallel where noted** — Most phases run sequentially. Phase 3 runs DEV and QA-PREPARE in parallel to save time.
5. **Shutdown after each phase** — Shut down each teammate before spawning the next phase's teammates.
6. **No user interaction** — All phases run autonomously EXCEPT Phase 6 (correct-course), which pauses for user decision.
7. **Dashboard rendering** — After completing each checkpoint (listed in DASHBOARD RENDERING), update the state file AND print the dashboard. Always read the state file first (in case of context compression), update it, write it back, then render from the updated state.
8. **Model selection** — Use model `sonnet` for all spawned teammates. Exception: use model `opus` for code review teammates (Phase 4 `reviewer-{n}`).

### Persistence Rules

1. **Before every update**: Read the file from disk. Do NOT assume in-memory values are current.
2. **After every update**: Write the full structure back to disk immediately.
3. **On phase start**: Set the phase status to `running`, set `teammate` to the teammate name, add teammate to `teammates.spawned` (if not already there) and `teammates.active`. Increment `teammates.total_count`.
4. **On phase end**: Set status to `done` (or `halted`/`warned`/`skipped`), remove teammate from `teammates.active`, update `details` with outcome summary.
5. **On convergence rounds**: Increment `rounds[0]` (current round), update `severity` array, set `convergence` to the detection result.
6. **Elapsed time**: Compute at render time by comparing current time to `start_time`. Do NOT store elapsed — compute it fresh each render.
7. **Issue accumulation**: `issues.total` reflects the LATEST known severity counts from the most recent convergence phase. Update when Phase 2 completes, Phase 4 completes (replaces Phase 2 counts), or Phase 5 completes.

### Recovery After Context Compression

If you lose context (you don't remember previous phases):
1. Read `_bmad-output/implementation-artifacts/pipeline-dashboard.yaml`
2. Find the last phase with status `done` — the NEXT phase is where you resume
3. If a phase has status `running` but no active teammate exists, treat it as interrupted — re-spawn the teammate for that phase
4. Continue the pipeline from the identified resume point

## DASHBOARD RENDERING

### When to Render

Print the dashboard at these checkpoints:

| # | Checkpoint | Trigger |
|---|-----------|---------|
| 1 | After SETUP | All phases initialized |
| 2 | Phase 1 done | CREATE-STORY complete |
| 3 | Each Phase 2 round | VALIDATE convergence check |
| 4 | Phase 3 start | Both DEV + QA-PREPARE spawned |
| 5 | Phase 3 partial | Either 3a or 3b completes first |
| 6 | Phase 3 done | Both 3a + 3b complete |
| 7 | Each Phase 4 round | CODE-REVIEW convergence check |
| 8 | Dev-fix done | After dev-fix sub-phase |
| 9 | Phase 5 done | QA-EXECUTE complete |
| 10 | Dev-bugfix done | After dev-bugfix sub-phase |
| 11 | Phase 5b done | QUALITY-GATE complete |
| 12 | Phase 6 trigger | CORRECT-COURSE triggered |
| 13 | COMPLETION | Final report REPLACES dashboard |

### Render Procedure

At each checkpoint:
1. Read `_bmad-output/implementation-artifacts/pipeline-dashboard.yaml`
2. Update the relevant phase/teammate/issue fields
3. Write the updated state back to the file
4. Compute elapsed time (see below)
5. Compute progress: count phases with status in {done, warned, skipped} / 8 total
6. Build the ASCII table from the state
7. Print the table

### Computing Elapsed Time

Run this bash command to get seconds elapsed:
```bash
echo $(( $(date +%s) - $(date -j -f "%Y-%m-%dT%H:%M:%SZ" "{start_time}" +%s) ))
```
Convert to human-readable:
- Under 60s: `{s}s`
- Under 3600s: `{m}m {s}s`
- Over 3600s: `{h}h {m}m`

### Status Display

| Internal Status | Display   |
|-----------------|-----------|
| `pending`       | `PENDING` |
| `running`       | `RUNNING` |
| `done`          | `DONE`    |
| `skipped`       | `SKIP`    |
| `halted`        | `HALTED`  |
| `warned`        | `WARN`    |

### Severity Display

Format severity arrays as: `{C}C {H}H {M}M {L}L`
Example: `0C 2H 5M 8L`
If all zeros, display: `clean`

### Progress Bar

40 characters wide. Filled: `=`, Empty: `.`
Format: `[====================....................] 50% (4/8 phases)`

### ASCII Dashboard Template

Print this structure at each checkpoint (adjust content per current state):

```
+-----------------------------------------------------------------+
|  AUTO-STORY PIPELINE: {story_key}                               |
|  Team: {team_name} | Elapsed: {elapsed}                        |
+--------------+-----------+--------+-------------+---------------+
| Phase        | Status    | Round  | Teammate    | Details       |
+--------------+-----------+--------+-------------+---------------+
| 1. Create    | {status}  | {r/m}  | {teammate}  | {details}     |
| 2. Validate  | {status}  | {r/m}  | {teammate}  | {severity}    |
| 3a. Dev      | {status}  | {r/m}  | {teammate}  | {details}     |
| 3b. QA Prep  | {status}  | {r/m}  | {teammate}  | {details}     |
| 4. Review    | {status}  | {r/m}  | {teammate}  | {severity}    |
| 5. QA Exec   | {status}  | {r/m}  | {teammate}  | {details}     |
| 5b. Gate     | {status}  | {r/m}  | {teammate}  | {gate_result} |
| 6. Correct   | {status}  | {r/m}  | {teammate}  | {details}     |
+--------------+-----------+--------+-------------+---------------+
| Progress: [{bar}] {pct}% ({done}/8 phases)                     |
| Active: {n} teammates | Spawned: {n} total | Issues: {severity}|
| Convergence: {last_convergence}                                 |
+-----------------------------------------------------------------+
```

**Formatting rules:**
- Use `+`, `-`, `|` for box drawing (plain ASCII, max compatibility)
- For pending phases, show `-` for teammate and `-/-` for round
- The convergence line shows the most recent event, e.g.:
  - `Convergence: Phase 2 clean after 2 rounds`
  - `Convergence: Phase 4 round 2/3 -- progress (2C 1H -> 1C 0H)`
  - `Convergence: none yet`
- Truncate teammate names at 11 chars with `..` if longer
- Truncate details at 13 chars with `..` if longer
- If pipeline is HALTED, add footer: `| ** PIPELINE HALTED -- see Phase {n} **`

## CONVERGENCE DETECTION

For Phases 2 and 4 (validation/review loops), track severity between rounds:

**After each round:**
1. Count issues by severity: CRITICAL, HIGH, MEDIUM, LOW
2. Compare with previous round's counts
3. Decide:

| Condition                                  | Action                              |
|--------------------------------------------|-------------------------------------|
| No CRITICAL or HIGH issues                 | **PROCEED** — clean                 |
| CRITICAL/HIGH count decreased vs last round | **RETRY** — progress being made     |
| CRITICAL/HIGH count same or increased       | **PROCEED + WARN** — stale loop     |
| CRITICAL security vulnerability + stale     | **HARD HALT** — report to user      |

**Hard cap**: Maximum 3 rounds per convergence loop. After round 3, proceed regardless with warnings logged.

**Dashboard**: Read state file. Set `phases.correct_course.status` = `running`. Set `pipeline.current_phase` = `correct_course`. Write state. **Render dashboard** (checkpoint 12).

**Action**: PAUSE the pipeline and present to the user:

```
## Architectural Violations Detected in {story-key}

The code review found the following architectural violations:
{list each violation with details}

**Options:**
1. Run /bmad-bmm-correct-course to formally address the issue
2. Proceed anyway (accept the violations as tech debt)
3. Stop the pipeline — I'll fix this manually
```

- If user chooses 1: Spawn teammate to run `/bmad-bmm-correct-course`, then resume from Phase 4
- If user chooses 2: Continue to Phase 5 with violations logged
- If user chooses 3: **HALT** pipeline

**Dashboard**: Set `phases.correct_course.status` = `done`/`skipped`/`halted`. Set details = user choice. Remove from active. Write state. **Render dashboard.**
