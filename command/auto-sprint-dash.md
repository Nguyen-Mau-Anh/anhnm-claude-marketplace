# BMAD Auto-Sprint Pipeline (Dashboard Edition)

You are the **sprint orchestrator**. Process ALL remaining stories in the current sprint by running the auto-story-dash pipeline for each one sequentially. You maintain a sprint-level dashboard that wraps the per-story dashboard.

---

## SETUP

1. **Read sprint-status.yaml** at `_bmad-output/implementation-artifacts/sprint-status.yaml`
   - If file not found: **HALT** — "No sprint-status.yaml found. Run /bmad-bmm-sprint-planning first."

2. **Build story list** — Parse all entries under `development_status:`
   - Separate epics (e.g., `epic-1`) from stories (e.g., `1-1-user-auth`)
   - Separate retrospectives (e.g., `epic-1-retrospective`)
   - Group stories by their epic

3. **Initialize sprint state** in `_bmad-output/implementation-artifacts/pipeline-dashboard.yaml`:
   - Record `start_time` by running: `date -u +%Y-%m-%dT%H:%M:%SZ`
   - Write the sprint state structure (see SPRINT STATE section)
   - Set each story's status based on sprint-status.yaml (`done` stories = `done`, others = `pending`)

4. **Render sprint dashboard** (shows all stories with their starting statuses)

---

## SPRINT STATE

Maintain sprint-level state in `_bmad-output/implementation-artifacts/pipeline-dashboard.yaml`. This file is shared with the per-story pipeline — the `sprint:` section persists while the `pipeline:` section resets for each story.

### Sprint State Schema

```yaml
# Auto-generated by auto-sprint-dash pipeline. Do not edit manually.
sprint:
  total_stories: 0
  stories_done: 0
  stories_halted: 0
  current_story: ""
  start_time: "{ISO 8601 timestamp}"
  stories:
    - key: "1-1-auth"
      epic: "epic-1"
      status: "done"
      details: "(was already done)"
    - key: "1-2-account"
      epic: "epic-1"
      status: "pending"
      details: "ready-for-dev"
    # ... one entry per story
  epics:
    - key: "epic-1"
      total: 3
      done: 0
    # ... one entry per epic
  quality_gates: { pass: 0, concerns: 0, fail: 0, waived: 0 }
  warnings: []

# Story-level state follows (managed by auto-story-dash logic)
pipeline:
  # ... (initialized per story, see auto-story-dash)
```

### Sprint State Rules

1. **Before each story**: Set `sprint.current_story`, update the story entry to `status: "running"`. Reset the `pipeline:` section for the new story.
2. **After each story**: Update the story entry with final status and details. Increment `sprint.stories_done` (or `stories_halted`). Update `sprint.quality_gates` counts. Update `sprint.epics` done counts.
3. **Always read-before-write** — same persistence rules as auto-story-dash.

## SPRINT DASHBOARD

### When to Render

| # | Checkpoint | Trigger |
|---|-----------|---------|
| 1 | After SETUP | All stories listed with starting statuses |
| 2 | Before each story | Story set to RUNNING, about to start pipeline |
| 3 | After each story | Story completed/halted, counts updated |
| 4 | After HARD HALT | Story halted, user decision pending |
| 5 | Epic completion | All stories in epic done |
| 6 | COMPLETION | Sprint report REPLACES dashboard |

### Sprint Dashboard Template

Print this structure at each sprint checkpoint:

```
+-----------------------------------------------------------------+
|  AUTO-SPRINT PIPELINE                                           |
|  Stories: {total} | Done: {done} | Elapsed: {elapsed}           |
+--------------+-----------+--------------------------------------+
| Story        | Status    | Details                              |
+--------------+-----------+--------------------------------------+
| 1-1-auth     | DONE      | (was already done)                   |
| 1-2-account  | DONE      | 7/8 phases, gate PASS 95%            |
| 1-3-plant    | RUNNING   | Phase 4 (Code Review) Round 1/3      |
| 2-1-persona  | PENDING   | backlog                              |
| 2-2-social   | PENDING   | backlog                              |
+--------------+-----------+--------------------------------------+
| Sprint: [{bar}] {pct}% ({done}/{total} stories)                |
| Epics: epic-1 (2/3 done) | epic-2 (0/3 done)                  |
| Gates: {p} PASS | {c} CONCERNS | {f} FAIL | {w} WAIVED         |
+-----------------------------------------------------------------+
```

**Formatting rules:**
- Same ASCII box-drawing as auto-story-dash (`+`, `-`, `|`)
- Progress bar: 40 chars, `=` filled, `.` empty, percentage based on stories
- Status values: `DONE`, `RUNNING`, `PENDING`, `HALTED`, `SKIP`
- For the RUNNING story, show current phase and round in Details
- Truncate story keys at 12 chars, details at 38 chars

### Combined Display

When a story is actively being processed, display BOTH dashboards:
1. Sprint dashboard first (compact overview)
2. Story dashboard below (detailed phase progress from auto-story-dash)

This gives the user both macro (sprint progress) and micro (story phase progress) views.

## EXECUTION

Process stories **in sprint-status.yaml order** (top to bottom):

### For EACH story (not epics, not retrospectives):

1. **Skip** if status == `done`
   - **Sprint dashboard**: Story already shows `DONE` with "(was already done)"

2. **Determine starting phase** from current status:
   | Status          | Starting Phase          |
   |-----------------|------------------------|
   | `backlog`       | Phase 1 (CREATE-STORY) |
   | `ready-for-dev` | Phase 3 (DEV-STORY)    |
   | `in-progress`   | Phase 3 (DEV-STORY)    |
   | `review`        | Phase 4 (CODE-REVIEW)  |

3. **Sprint dashboard update**: Set the story to `status: "running"`, set `sprint.current_story` = story key. Update story details to show starting phase. **Render sprint dashboard.**

4. **Run story pipeline** — Follow ALL instructions from `/auto-story-dash` for this story (model selection: all spawned teammates use `sonnet`, except code review teammates which use `opus`):
   - Read `.claude/commands/auto-story-dash.md` and execute its full pipeline
   - The story-level `pipeline:` section in the state file is initialized fresh for this story
   - The sprint-level `sprint:` section persists across stories
   - All 13 story dashboard checkpoints render the story dashboard (below the sprint dashboard)
   - When the story pipeline hits COMPLETION, control returns here

5. **Sprint dashboard update**: Read state file. Update the story entry:
   - Set `status` = `done` or `halted`
   - Set `details` = summary from story pipeline (e.g., "8/8 phases, gate PASS 95%")
   - Increment `sprint.stories_done` (or `sprint.stories_halted`)
   - Update `sprint.quality_gates` based on story's gate result
   - Update the epic's done count
   - Write state file. **Render sprint dashboard.**

6. **Handle outcomes:**
   - **SUCCESS**: Continue to next story automatically
   - **HARD HALT**: Present to user:
     ```
     Story {story-key} encountered a HARD HALT: {reason}
     1. Continue to next story (skip this one)
     2. Stop the sprint pipeline
     ```

7. **Epic completion check**: After processing all stories in an epic:
   - If ALL stories in the epic are `done`: update epic status to `done` in sprint-status.yaml
   - **Sprint dashboard**: Update epic display. **Render sprint dashboard.**
   - If epic is `done` AND retrospective is `optional`:
     ```
     Epic {epic-id} is complete. Run retrospective? (optional)
     1. Skip retrospective
     2. Run /bmad-bmm-retrospective for this epic
     ```

## COMPLETION

After processing all stories (or user stops):

1. **Render final sprint dashboard** — one last render showing all stories completed

2. **Archive state file**: Rename `pipeline-dashboard.yaml` to `pipeline-dashboard-sprint-done.yaml`

3. **Print sprint completion report** (replaces dashboard):

```

| 1-2-account-mgmt  | done        | 5/7    | PASS (95%)   | 2 medium  | none            |
| 1-3-plant-data    | done        | 7/7    | PASS (100%)  | 0         | none            |
| 2-1-personality   | HALTED      | 3/7    | --           | 1 critical | security issue  |
```
